MAA Unified Application - Developer Guide

Version 1.05 • March 26, 2026 • ~30k LOC Production App

1. Introduction & Purpose

The MAA Unified Application is a production-ready Flask + SocketIO web platform for Oracle Maximum Availability Architecture (MAA) teams. It provides a single pane of glass for managing Exadata racks, including:

  • Environment setup (SNMP, ASR, SMTP, passwords via CellCLI/DBMCLI/ILOM)
  • Coordinated patching of Storage Servers, Database Servers, and Guest VMs using patchmgr
  • Exascale monitoring setup (excmonitor user + RSA PKI wallet)
  • Oracle Enterprise Manager agent lifecycle (install, status, cleanup, Falcon sensor)
  • Persistent job scheduling with APScheduler + Oracle jobstore
  • Real-time execution logging with browser tabs
  • Credential vault, reachability checks, and topology discovery (racks, clusters, ILOMs, switches)
Key Exadata Domain Constraints Handled: ≤8 DB servers + ≤14 storage servers per rack, majority-up reachability rule, default users (root/oracle/maamd/admin), physical vs virtual detection.

10. Adding New Functions (Extension Guide)

  1. Edit functions dict in environment_setup_functions.py
  2. Add display_name to get_functions_for_type() map
  3. If special: implement in the same file or new run_*_background
  4. (Optional) Extend get_component_type() for new type
  5. UI auto-updates via functions_by_type
  6. Add to SocketIO handler if group-level
  7. Update maamd_schema.sql only if new metadata needed
  8. Test with reachability + credential lookup

Example (add Falcon sensor):

('install_falcon_sensor', 'special_python_function', 'Install/Upgrade CrowdStrike')

11. Running, Deployment & Debugging

python maa_unified_app.py --host 0.0.0.0 --port 6003

Required env vars: DB_PASSWORD, FLASK_SECRET_KEY, CONFLUENCE_TOKEN

Logs: output/maa_unified.log + browser tabs. Instance lock: /tmp/maa_unified_app_new.lock

12. Troubleshooting

  • Lock conflict: DELETE FROM MAAMD.JOB_LOCKS WHERE IS_RUNNING='Y'
  • Patchmgr missing: Verify TEST_CONFIG_SCRIPTS/PATCHING/dbserver_patch_XX/ exists + chmod 755 patchmgr
  • SocketIO no logs: Pass correct client_sid in form POST
  • Credential fail: Check ACCESS_CREDENTIALS BLOBs via decrypt test
  • Exascale discovery fail: Provide manual rest_endpoint
  • Reachability false negative: Use TCP/22 check (firewall-friendly)

Common Edge Cases: Empty clusters → default creds, hypervisor detection via /EXAVMIMAGES, majority-up rule.
